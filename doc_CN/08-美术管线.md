# 美术管线

美术管线从创意方向到最终素材产出的完整流程。

## 管线流程

```
art-director（方向）
    ↓ 美术圣经 + 视觉规则
art-bible Skill（美术圣经）
    ↓ design/art/art-bible.md
asset-spec Skill（素材规格）
    ↓ design/assets/specs/[系统].md
art-producer Agent（素材产出）
    ↓ assets/art/ 目录下的实际素材
dev-story Skill（代码引用）
    ↓ 使用素材路径常量引用素材
```

---

## 各环节详解

### 1. art-director — 美术总监

**角色**：定义视觉方向，不产出实际像素/3D 素材。

**职责**：
- 创建和维护美术圣经
- 定义调色板、形状语言、视觉规则
- 审查素材质量和风格一致性
- 委派给 `art-producer` 产出实际素材
- 委派给 `technical-artist` 处理着色器/VFX

**不做的**：直接创建像素画/3D 模型（委派给 `art-producer`）。

---

### 2. `/art-bible` — 美术圣经

**触发时机**：概念阶段确定后、系统设计之前（Drive 流程中的 C.3b 步骤）。

**产出**：`design/art/art-bible.md`

**包含内容**：
- 视觉方向声明
- 调色板（主色、辅色、强调色、含义）
- 形状语言（尖锐/圆润/有机/几何）
- 角色设计方向
- 环境设计方向
- UI 视觉规范
- 素材命名规范
- 文件大小预算

**门禁**：AD-ART-BIBLE — 美术圣经完成后由 art-director 审查签核。

---

### 3. `/asset-spec` — 素材规格

**触发时机**：技术准备阶段，为每个有视觉需求的 MVP 系统生成。

**产出**：`design/assets/specs/[系统名].md`

**包含内容**：
- 需要的素材清单（精灵/模型/纹理/动画/VFX）
- 每个素材的尺寸/分辨率要求
- 文件格式和命名规范
- 动画帧数和时间
- 色板引用
- 技术约束（多边形预算、纹理大小等）

**命名规范**：`[分类]_[名称]_[变体]_[尺寸].[扩展名]`

---

### 4. art-producer — 美术素材产出

**角色**：根据规格调用外部 API 生成实际美术素材。

**支持的 API**：

| API | 提供方 | 环境变量 | 特点 |
|-----|--------|---------|------|
| **阿里云百炼万相2.6** | Alibaba Cloud | `DASHSCOPE_API_KEY` | 中国用户首选，支持中文提示词，多种尺寸，异步调用 |
| **OpenAI DALL-E 3** | OpenAI | `OPENAI_API_KEY` | 高质量英文生成 |
| **Stability AI** | Stability | `STABILITY_API_KEY` | 风格化图像，细粒度控制 |
| **Local SD WebUI** | 自部署 | 无需 | 本地运行，无费用 |
| **ComfyUI** | 自部署 | 无需 | 自定义工作流 |

**首次运行引导**：art-producer 首次被调用时，如果 `design/art/api-config.md` 不存在，会引导用户选择 API 并配置。API Key 存储在环境变量中，不写入文件。

**工作流程**：
1. 读取素材规格文件
2. 检查 API 配置（`design/art/api-config.md`），未配置则引导用户
3. 根据美术圣经调色板和风格构造提示词
4. 调用 API 生成素材（百炼使用异步模式，轮询直到完成）
5. 下载并验证输出质量（色板一致性、尺寸正确性）
6. 保存到 `assets/art/` 对应目录
7. 更新 `design/assets/asset-manifest.md` 状态为 `Produced`

**百炼万相2.6 调用模式**：
- 文生图模式（`enable_interleave=true`）：纯文本描述生成图像，适合游戏素材
- 图像编辑模式（`enable_interleave=false`）：基于参考图修改，适合风格迁移
- 异步调用：创建任务 → 轮询状态（10 秒间隔）→ 下载结果（24 小时内有效）

**素材目录结构**：
```
assets/art/
├── characters/     # 角色素材
├── environments/   # 环境素材
├── items/          # 物品图标
├── ui/             # UI 元素
├── vfx/            # 特效帧
└── textures/       # 纹理
```

### 5. 集成到开发流程

美术管线在各 Drive 自动化路线中的集成点：

#### `/drive` 和 `/drive-init`

| 步骤 | 动作 |
|------|------|
| C.3b | 创建美术圣经 |
| T.3b | 为 MVP 系统生成素材规格（逐个循环） |
| Pr.2b / I.20b | 并行产出美术素材（与代码实现同步） |

#### `/drive-add`

| 步骤 | 动作 |
|------|------|
| A.4b | 新系统 GDD 审批后生成素材规格 |
| A.11b | 实现时并行产出美术素材 |

#### `/drive-iterate`

| 步骤 | 动作 |
|------|------|
| M.10b | 视觉内容变化时更新素材规格 |
| M.10c | 重新生成受影响的美术素材 |

## 已知缺口：素材引用链

**当前状态**：`art-producer` 生成素材但 `dev-story` 没有机制确保代码正确引用。

**计划中的解决方案**：

1. **art-producer**：生成素材后自动产出引擎特定的常量文件
   - Godot: `src/autoload/asset_registry.gd`
   - Unity: `Assets/Scripts/AssetPaths.cs`


2. **dev-story**：实现前读取素材注册表，用常量 ID 引用路径

3. **smoke-check**：添加路径存在性验证 — 扫描代码中的素材引用是否指向实际文件

4. **coding-standards**：添加禁止模式："禁止硬编码素材路径，必须通过常量引用"

**优先级**：高，延迟到实际项目开始时实现。

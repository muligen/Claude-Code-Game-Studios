# CCGS 框架批判性分析

> 综合多轮批判讨论，按优先级排列。每个问题附解决方案和验证标准。

---

## P0 — 阻塞性问题（不修就无法使用）

### 1. Skill 间接口未验证对齐

**问题**：77 个 Skill 像流水线上的工位，但没人验证过上一个工位的产出格式是否匹配下一个工位的输入要求。`/brainstorm` 产出的 game-concept.md 能被 `/map-systems` 正确解析吗？`/design-system` 写的 GDD 包含 `/dev-story` 需要的 TR-ID 吗？这些接口契约只存在于设想中。

**影响**：用户跑到第三四个 Skill 就会因为格式不匹配而卡住，且错误信息不清晰——看起来是 Skill 本身坏了，实际是上游产出格式变了。

**解决方案**：跑一次完整管线（drive-init 或手动顺序调用核心 Skill），在每个衔接点记录实际输入/输出格式，用实际数据修正 Skill 定义。

**验证标准**：连续调用 brainstorm → map-systems → design-system → dev-story，中间不需要手动干预格式。

---

### 2. 无引擎编辑器集成

**问题**：框架假设开发者在 CLI 里完成所有工作，但真实的 Godot/Unity 开发在编辑器里进行——拖节点、调参数、跑场景、看效果。框架和引擎编辑器之间是空白的。开发者通过 `/dev-story` 生成代码文件后，要手动切到编辑器导入、运行、调试。

**影响**：管线在最关键的环节——"看到你做的东西"——断了。每次代码变更都需要人工桥接 CLI 和编辑器。

**解决方案**：
- 加 `/run` Skill：通过命令行启动引擎并运行当前场景（Godot: `godot --path . --debug`）
- 加 `/build` Skill：生成独立构建
- 加 `/iterate` Skill：运行 → 捕获输出/截图 → 定向修改 → 重运行
- 在 `/dev-story` 完成后自动提示用户运行游戏验证

**验证标准**：实现一个 Story 后，一条命令能在引擎里看到运行结果。

---

### 3. AI 产出无格式校验

**问题**：TODO.md 的乱码是 AI 产出不可靠的证据。框架把所有 AI 产出当作可靠数据——设计文档喂给代码生成，状态文件驱动 Drive 循环，Story 验收标准决定是否推进。如果 AI 在简单任务（写 TODO）上都能产出乱码，在复杂任务（写 GDD、架构文档）上也会。但框架没有任何格式校验机制。

**影响**：一个损坏的 `active.md` 可以让 Drive 循环走向错误方向。一个格式异常的 GDD 可以让 `/dev-story` 产出错误代码。错误会沿管线级联放大。

**解决方案**：
- 关键文件加 schema 校验（GDD 必须有 8 个章节、Story 必须有 TR-ID、状态文件必须包含特定字段）
- `/gate-check` 的 artifact 检查增加格式验证（不只检查文件存在，还检查结构完整）
- Session hooks 里加 `validate-state.sh`：在关键操作后自动校验状态文件格式

**验证标准**：故意给一个 GDD 缺少必需章节，框架能检测并报错，而不是默默继续。

## P1 — 严重问题（可用但体验差）

### 4. 成本不透明

**问题**：一次完整的 `/drive-init` 25 步，每步 2-3 次 Task 调用，每次 10-30K token，总计 500K-2M token，可能花费 $15-60。用户在跑到第 15 步时才发现成本已经很高了。框架没有 token 预算概念。

**解决方案**：
- 每个 Gate 之前估算当前会话已消耗的 token 量（通过文件产出量近似估算）
- 加 `solo` 模式的轻量路径：跳过非必要文档生成，直接从概念到代码
- 在 `/drive-init` 开头显示预估成本范围："完整模式约 $X-Y，快速模式约 $A-B"

**验证标准**：用户在开始前知道大概要花多少，跑到一半时知道还剩多少预算。

---

### 5. 部分 Agent 定义差异化不足

**问题**：50 个 Agent 的数量不是问题——按需调用、独立上下文窗口、不同 prompt 确实产生不同视角。问题在于部分 Agent 的边界模糊，导致调用时不知道该选谁：

- `systems-designer` vs `game-designer`：systems-designer 做"具体机制设计、公式"，game-designer 也包含"mechanics, systems, balancing"。两者被分配同一任务时产出无本质区别。
- `economy-designer` vs `game-designer`：game-designer 描述已包含"economy, balancing"，economy-designer 是从它拆出的子领域。
- `writer` vs `world-builder`：边界主要靠职责描述，缺乏独特的方法论约束。

**解决方案**：
- 每个 Agent 定义增加 "Unique Methodology" 段：该 Agent 使用什么独特的思考框架、输出格式、或质量标准
- 明确重叠领域的路由规则：什么时候找 game-designer，什么时候找 systems-designer
- 在 agent-roster.md 中标注每个 Agent 的"不可替代场景"——什么时候必须用它而不是通用的

**验证标准**：给同一个设计问题分别调用两个易混淆的 Agent，产出有明显不同的视角和重点。

---

### 6. 上下文加载效率低

**问题**：`/drive` 的 Phase 1 每次迭代读 15+ 文件，单次循环消耗 30-50K token。3-4 轮后上下文满了。77 个 Skill 文件共 883KB，不可能同时加载。

**解决方案**：
- 加 `/drive status` 时生成压缩版 `project-snapshot.md`（单文件摘要当前项目状态）
- Drive 循环的 Phase 1 改为读 `project-snapshot.md` 而非 15 个原始文件
- 在关键节点（gate-check 后、sprint 结束后）自动更新 snapshot

**验证标准**：Drive 循环单次迭代的 token 消耗降低 50%+，连续跑 8 轮不需要 compact。

---

### 7. 缺少参考项目

**问题**：产品是管线工具本身，不是游戏。但管线工具需要证明它能跑通——Unreal Engine 带模板项目，React 带教程 App。CCGS 没有任何端到端示例，用户拿到手第一步就是踩进未知坑里。

**解决方案**：创建一个参考项目（如 Godot 2D 平台跳跃），包含：
- 每个阶段的产物快照（concept → GDD → ADR → code → test）
- 一个可以 `git clone` 下来直接跑的 Godot 项目
- 标注每个文件是由哪个 Skill 产出的

**验证标准**：新用户看完参考项目后，能理解管线的输入/输出并开始自己的项目。

## P2 — 改进项（可用，做了更好）

### 8. 中英文档双重维护负担

**问题**：`doc_CN/` 10 份中文文档 + 英文原文。每次改 Skill 需要同步更新两份文档。要么很快过时，要么成为持续负担。

**解决方案**：
- 短期：在中文文档头注明"最后同步日期"，过时即标记
- 中期：关键文档（概览、阶段、规范）保持中英双语，详细清单型文档（Skill 清单、Agent 清单）只保留中文版，链接英文源文件
- 长期：考虑用脚本从英文 Skill/Agent 定义自动生成清单型文档

**验证标准**：改一个 Skill 后，5 分钟内能完成文档同步。

---

### 9. 50 个 Agent 的维护成本

**问题**：Godot 4.7 改 API → 更新 4 个 Godot Agent。Claude Code 改 Task 行为 → 可能影响所有 Skill。模型行为变化 → 精心调优的 Agent prompt 可能失效。50 个 Agent = 50 份维护负债。

**解决方案**：
- 给每个 Agent 加版本标记和最后验证日期
- 用 `/skill-test` 扩展成 agent-test，自动验证 Agent 产出质量
- 按使用频率分级维护：核心 Agent（8个）每次发布前验证，边缘 Agent 按需验证

**验证标准**：引擎大版本更新后，一周内完成所有相关 Agent 的验证和修复。

---

### 10. 框架复制人类工作流，未探索 AI 原生工作流

**问题**：架构直接映射人类工作室（设计→程序→美术→QA）。但 AI 的工作方式可能需要不同组织方式：也许不需要"先写 GDD 再写代码"——AI 可以同时产出设计和实现；也许不需要独立的"代码审查"——可以在生成时就内建约束。

**解决方案**：这个不需要现在解决，但值得在 v2 时探索：
- 尝试"AI 原型模式"：从概念直接到可玩原型，跳过所有文档
- 尝试"同步生成模式"：一个 Agent 调用同时产出 GDD 段落 + 实现代码
- 收集用户反馈：哪些"人类工作流步骤"在 AI 辅助下是多余的

**验证标准**：v2 有至少一种"AI 原生工作流"作为实验性选项。

## 优先级总结

| 优先级 | # | 问题 | 一句话描述 | 预估工作量 |
|--------|---|------|-----------|-----------|
| **P0** | 1 | Skill 接口对齐 | 上游产出格式和下游输入格式不匹配 | 中 |
| **P0** | 2 | 引擎编辑器集成 | CLI 产出的代码无法一键运行验证 | 中 |
| **P0** | 3 | AI 产出格式校验 | 损坏的状态文件/文档会导致级联错误 | 中 |
| **P1** | 4 | 成本透明 | 用户不知道一次完整管线跑下来花多少 | 小 |
| **P1** | 5 | Agent 差异化 | 部分 Agent 边界模糊，调用时不知道选谁 | 中 |
| **P1** | 6 | 上下文效率 | Drive 循环 3-4 轮就爆上下文 | 中 |
| **P1** | 7 | 参考项目 | 新用户没有端到端示例参考 | 大 |
| **P2** | 8 | 双语文档维护 | 改一处要同步两份文档 | 小 |
| **P2** | 9 | Agent 维护成本 | 50 个 Agent 跟不上引擎/模型更新 | 小 |
| **P2** | 10 | AI 原生工作流 | 当前架构照搬人类工作流，可能非最优 | v2 探索 |

## 建议执行顺序

```
P0-1 接口对齐 ←── 最先做，一边跑一边记，跑完就知道了
    ↓
P0-3 格式校验 ←── 跑接口对齐时自然会发现哪些点需要校验
    ↓
P0-2 引擎集成 ←── 接口通了对齐后，加运行能力
    ↓
P1-7 参考项目 ←── P0 全修好后，跑一次完整管线，产物就是参考项目
    ↓
P1-4/5/6       ←── 有了参考项目，这些优化有具体数据支撑
P2-8/9/10      ←── 长期维护项
```

# 自动化管线

4 条 Drive 自动化路线，用户只需在每步回答 yes/no，系统自动推进。

## 共享的 Drive Loop 结构

所有 Drive 管线都遵循相同的四阶段循环：

```
┌──────────────────────────────────────┐
│  Phase 1: Assess（评估）             │
│  读取状态文件、扫描项目产物           │
│            ↓                         │
│  Phase 2: Propose（提议）            │
│  展示下一步动作，等待 yes/no         │
│            ↓                         │
│  Phase 3: Execute（执行）            │
│  委派给 Skill 或 Agent，验证产出     │
│            ↓                         │
│  Phase 4: Record（记录）             │
│  更新状态文件，回到 Phase 1          │
└──────────────────────────────────────┘
```

**用户交互**：每步提供四个选项：
- **Yes, proceed** — 执行
- **Not now** — 延后
- **Skip** — 永久跳过
- **Why this step?** — 查看原因后再决定

## 1. `/drive` — 项目自动巡航

**适用场景**：已有项目，从当前状态自动推进到发布。

**完整路线**（7 个阶段全覆盖）：

| 阶段 | 步骤 |
|------|------|
| Concept | C.1 头脑风暴 → C.2 审查 → C.3 支柱 → C.3b 美术圣经 → C.4 门禁 |
| Systems Design | S.1 映射系统 → S.2 设计 GDD（逐个循环）→ S.3 审查 → S.4 门禁 |
| Technical Setup | T.1 配置引擎 → T.2 ADR → T.3 补充 ADR → T.3b 素材规格 → T.4 门禁 |
| Pre-Production | P.1 原型 → P.2 测试 → P.3 Go/No-Go → P.4 补 GDD → P.5 里程碑 → P.6 Sprint → P.7 门禁 |
| Production | Pr.1 Sprint 规划 → Pr.2 实现（逐个 Story）→ Pr.2b 美术产出 → Pr.3 审查 → Pr.4 测试 → Pr.5 回顾 → ... → Pr.9 门禁 |
| Polish | Po.1 性能 → Po.2 平衡 → Po.3 素材审计 → Po.4 打磨 → Po.5 无障碍 → Po.6 本地化 → Po.7 测试 → Po.8 门禁 |
| Release | R.1 检查清单 → R.2 上线清单 → R.3 变更日志 → R.4 补丁说明 → R.5 执行 → R.6 监控 |

**特殊模式**：
- `/drive status` — 查看当前位置
- `/drive reset` — 重新评估
- `/drive [阶段名]` — 跳转到指定阶段

**状态文件**：`production/session-state/drive-state.md`

## 3. `/drive-iterate` — 修改已有设计

**适用场景**：修改游戏中已有的系统/机制/数值。

**核心逻辑**：用户描述修改意图 → 自动分析影响面 → 判断改动级别 → 执行对应流程 → 验证无回归。

### 改动级别自动判断

| 级别 | 判断条件 | 示例 |
|------|---------|------|
| **minor** | 只改数据/配置值，不改规则 | 伤害 10→15，刷新率 0.5→0.3 |
| **moderate** | 单个系统内规则/机制变化 | 耐力回复公式修改，新增状态效果 |
| **major** | 系统间交互变化，新增/删除系统 | 战斗全面重构，添加多人模式 |

用户可在自动判断后覆盖级别。

### 流程（14 步）

| ID | 动作 | 适用级别 |
|----|------|---------|
| M.1 | 收集修改意图 | 全部 |
| M.2 | 范围分析 | 全部 |
| M.3 | 自动判断改动级别 | 全部 |
| M.4 | 影响面分析 | moderate/major |
| M.5a | 快速设计规格 | minor |
| M.5b | 更新 GDD | moderate |
| M.5c | 重写 GDD + 新 ADR | major |
| M.8 | 跨 GDD 一致性检查 | moderate/major |
| M.9 | 更新关联 Story/ADR | moderate/major |
| M.10 | 实现代码变更 | 全部 |
| M.11 | 验收 Story | 全部 |
| M.12 | 回归测试 | moderate/major |
| M.13 | 数值平衡检查 | 涉及公式/数据时 |
| M.14 | Smoke check | 全部 → **DONE** |

**minor 路径可选**：用户可选择跳过一致性检查、回归测试等，仅做 Smoke check。

**状态文件**：`production/session-state/drive-iterate-state.md`

## 4. `/drive-add` — 添加新设计

**适用场景**：向游戏中添加全新的系统/功能/内容。

### 流程（14 步）

| ID | 动作 | 产出物 |
|----|------|--------|
| A.1 | 收集新设计意图 | 功能描述 |
| A.2 | 更新系统索引 | 更新的 systems-index.md |
| A.3 | 设计新系统 GDD | 新 GDD |
| A.4 | 审查新 GDD | 审查结论 |
| A.4b | 生成素材规格 | `design/assets/specs/` |
| A.5 | 如需新 ADR | 新 ADR |
| A.6 | 跨 GDD 一致性检查 | 冲突报告 |
| A.7 | 跨 GDD 全局审查 | 审查报告 |
| A.8 | 创建 Epic | Epic 文件 |
| A.9 | 拆分 Story | Story 文件 |
| A.10 | 排入 Sprint | Sprint 计划 |
| A.11 | 实现 Story（逐个循环） | 代码 + 测试 |
| A.11b | 并行产出美术素材 | 美术素材 |
| A.12 | 验收 Story | Story Complete |
| A.13 | 内容审计 | 缺口报告 |
| A.14 | Smoke check | PASS → **DONE** |

**特殊情况**：
- 新系统依赖未实现的系统 → 提示先实现依赖项
- 新系统与现有 ADR 冲突 → 标记并解决
- 添加的是内容而非新系统 → 走轻量流程（跳过 A.2、A.5）

**状态文件**：`production/session-state/drive-add-state.md`

## 共通机制

### 门禁 FAIL 处理

任何门禁返回 FAIL：
1. 读取门禁输出，识别失败项
2. 逐一提出针对性修复步骤
3. 修复后重新提议门禁检查
4. 不自动跳过直到 PASS

### 用户中断

用户在循环中输入非 yes/no 的自由请求：
1. 直接处理请求（或委派给对应 Agent）
2. 处理完毕后更新状态文件
3. 从 Phase 1 重新评估继续循环

### 上下文窗口压力

对话过长时：
1. 确保状态文件完全更新
2. 建议运行 `/compact Focus on the Drive Loop — state is in production/session-state/drive-state.md`
3. 压缩后 Drive Loop 从状态文件恢复

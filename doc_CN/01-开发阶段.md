# 开发阶段流程

CCGS 将游戏开发划分为 7 个阶段，每个阶段有明确的产物要求和门禁检查。

## 阶段一览

```
Concept → Systems Design → Technical Setup → Pre-Production → Production → Polish → Release
```

阶段间通过 `/gate-check` 强制验证，确保质量后才推进。

---

## 1. Concept（概念阶段）

**目标**：确定游戏核心概念、设计支柱和美术方向。

| 步骤 | 动作 | 命令/Agent | 产出物 |
|------|------|-----------|--------|
| C.1 | 头脑风暴游戏概念 | `/brainstorm` | `design/gdd/game-concept.md` |
| C.2 | 审查概念质量 | `/design-review` | 审查结论 |
| C.3 | 定义游戏支柱 | `creative-director` | `design/gdd/game-pillars.md` |
| C.3b | 创建美术圣经 | `/art-bible` | `design/art/art-bible.md` |
| C.4 | 门禁检查 | `/gate-check systems-design` | PASS/CONCERNS/FAIL |

**关键产出**：游戏概念文档、设计支柱、反支柱、美术圣经。

---

## 2. Systems Design（系统设计阶段）

**目标**：将概念拆解为独立系统，为每个系统编写完整的游戏设计文档（GDD）。

| 步骤 | 动作 | 命令/Agent | 产出物 |
|------|------|-----------|--------|
| S.1 | 映射所有系统和依赖关系 | `/map-systems` | `design/gdd/systems-index.md` |
| S.2 | 逐个设计系统 GDD | `/design-system [系统名]` | 每个 GDD 文件 |
| S.3 | 审查每个 GDD | `/design-review [路径]` | 审查结论 |
| S.4 | 门禁检查 | `/gate-check technical-setup` | PASS/CONCERNS/FAIL |

**系统索引包含**：
- 所有系统的名称和职责
- 系统间的依赖关系图
- 优先级分层（MVP / 垂直切片 / Alpha / 完整愿景）

**GDD 必须包含的 8 个部分**：概述、玩家幻想、详细规则、公式、边界情况、依赖、调优旋钮、验收标准。

---

## 3. Technical Setup（技术准备阶段）

**目标**：配置引擎、建立架构、为实现做好准备。

| 步骤 | 动作 | 命令/Agent | 产出物 |
|------|------|-----------|--------|
| T.1 | 配置引擎和偏好 | `/setup-engine` | CLAUDE.md + technical-preferences |
| T.2 | 创建初始架构决策 | `/architecture-decision` | 第一个 ADR |
| T.3 | 核心系统补充 ADR | `technical-director` | 更多 ADR |
| T.3b | 为 MVP 系统生成素材规格 | `/asset-spec system:[系统]` | `design/assets/specs/` |
| T.4 | 门禁检查 | `/gate-check pre-production` | PASS/CONCERNS/FAIL |

**关键产物**：
- `docs/architecture/architecture.md` — 主架构文档
- `docs/architecture/` — 架构决策记录（ADR）
- `docs/architecture/control-manifest.md` — 程序员规则清单
- `.claude/docs/technical-preferences.md` — 技术偏好配置

---

## 4. Pre-Production（预生产阶段）

**目标**：验证核心机制可行性，创建 Epic 和 Sprint 计划。

| 步骤 | 动作 | 命令/Agent | 产出物 |
|------|------|-----------|--------|
| P.1 | 原型验证核心机制 | `/prototype [核心机制]` | 原型 + 报告 |
| P.2 | 原型测试 | `/playtest-report` | 测试报告 |
| P.3 | Go/No-Go 决策 | `creative-director` | 决策记录 |
| P.4 | 补完剩余 GDD | `/design-system` | 补充的 GDD |
| P.5 | 创建里程碑 | `producer` | 里程碑文件 |
| P.6 | 规划首个 Sprint | `/sprint-plan new` | Sprint 计划 |
| P.7 | 门禁检查 | `/gate-check production` | PASS/CONCERNS/FAIL |

**Go/No-Go 决策失败时**：回到 P.1 调整原型或重新设计核心机制。

---

## 5. Production（生产阶段）

**目标**：通过 Sprint 循环实现所有系统。

### Sprint 循环

```
Sprint 规划 → 任务实现 → 代码审查 → 测试 → Sprint 回顾 → 下一个 Sprint
```

| 步骤 | 动作 | 命令/Agent |
|------|------|-----------|
| Pr.1 | Sprint 规划 | `/sprint-plan` |
| Pr.2 | 逐个实现任务 | `/dev-story` 或 `team-*` |
| Pr.2b | 并行产出美术素材 | `art-producer` |
| Pr.3 | 代码审查 | `/code-review` |
| Pr.4 | 功能测试 | `qa-tester` |
| Pr.5 | Sprint 回顾 | `/retrospective` |
| Pr.6 | 里程碑审查 | `/milestone-review` |
| Pr.7 | 范围检查（每 2-3 Sprint） | `/scope-check` |
| Pr.8 | 技术债扫描（每 2-3 Sprint） | `/tech-debt` |
| Pr.9 | 门禁检查 | `/gate-check polish` |

**关键特点**：
- Pr.2 逐个 Story 实现，每个都有批准-执行周期
- Pr.2b 与代码实现并行进行，仅在有视觉需求的 Story 触发
- 任何门禁 FAIL → 自动提出修复步骤，修复后重试

## 阶段自动检测

如果 `production/stage.txt` 不存在，框架通过产物自动推断当前阶段：

| 阶段 | 判断条件 |
|------|---------|
| Concept | 无 `game-concept.md` |
| Systems Design | 有概念文档，无或未完成 `systems-index.md` |
| Technical Setup | 有系统索引，引擎未配置 |
| Pre-Production | 引擎已配置，`src/` 少于 10 个文件 |
| Production | `src/` 有 10+ 文件，活跃开发中 |
| Polish / Release | 仅通过 `/gate-check` 显式设置 |

使用 `/project-stage-detect` 可随时手动检测。

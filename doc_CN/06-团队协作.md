# 团队协作

## Team 编排技能

9 个 Team 技能用于协调多个 Agent 完成跨领域的任务。

| 技能 | 协调的 Agent | 适用场景 |
|------|-------------|---------|
| `/team-combat` | game-designer + gameplay-programmer + ai-programmer + technical-artist + sound-designer + qa-tester | 战斗系统：设计 → AI → 特效 → 音效 → 测试 |
| `/team-narrative` | narrative-director + writer + world-builder + level-designer | 故事内容：方向 → 对话 → 世界观 → 关卡叙事 |
| `/team-ui` | ux-designer + ui-programmer + art-director + accessibility-specialist | 界面系统：交互设计 → 实现 → 视觉 → 无障碍 |
| `/team-release` | release-manager + qa-lead + devops-engineer + producer | 发布流程：检查 → 质量 → 构建 → 协调 |
| `/team-polish` | performance-analyst + technical-artist + sound-designer + qa-tester | 打磨优化：性能 → 视效 → 音效 → 质量 |
| `/team-audio` | audio-director + sound-designer + technical-artist + gameplay-programmer | 音频系统：方向 → 音效 → 视觉反馈 → 代码集成 |
| `/team-level` | level-designer + narrative-director + world-builder + art-director + systems-designer + qa-tester | 关卡制作：布局 → 叙事 → 世界 → 视觉 → 机制 → 测试 |
| `/team-live-ops` | live-ops-designer + economy-designer + community-manager + analytics-engineer | 在线运营：活动设计 → 经济 → 社区 → 数据 |
| `/team-qa` | qa-lead + qa-tester + gameplay-programmer + producer | 质量保障：策略 → 执行 → 修复 → 协调 |

## 常见工作流模式

### 模式 1：新功能（全流程）

```
1. creative-director  — 审批功能概念
2. game-designer      — 创建设计文档
3. producer           — 排期、识别依赖
4. lead-programmer    — 设计代码架构
5. [专家程序员]        — 实现功能
6. technical-artist   — 实现视觉特效
7. writer             — 创建文本内容
8. sound-designer     — 创建音频事件
9. qa-tester          — 编写测试用例
10. qa-lead           — 审查测试覆盖
11. lead-programmer   — 代码审查
12. qa-tester         — 执行测试
13. producer          — 标记完成
```

### 模式 2：Bug 修复

```
1. qa-tester          — 提交缺陷报告（/bug-report）
2. qa-lead            — 分流严重度和优先级
3. producer           — 分配到 Sprint
4. lead-programmer    — 定位根因、分配程序员
5. [专家程序员]        — 修复
6. lead-programmer    — 代码审查
7. qa-tester          — 验证修复 + 回归测试
8. qa-lead            — 关闭缺陷
```

### 模式 3：数值平衡调整

```
1. analytics-engineer — 从数据发现不平衡
2. game-designer      — 对照设计意图评估
3. economy-designer   — 建模调整方案
4. game-designer      — 审批新数值
5. [更新数据文件]      — 修改配置
6. qa-tester          — 回归测试受影响系统
7. analytics-engineer — 监控变更后指标
```

### 模式 4：新关卡/区域

```
1. narrative-director — 定义区域叙事目的
2. world-builder      — 创建背景故事和环境上下文
3. level-designer     — 设计布局、遭遇、节奏
4. game-designer      — 审查遭遇的机制设计
5. art-director       — 定义区域视觉方向
6. audio-director     — 定义区域音频方向
7. [相关程序员和美术]  — 实现
8. writer             — 创建区域文本
9. qa-tester          — 测试完整区域
```

### 模式 5：快速原型

```
1. game-designer      — 定义假设和成功标准
2. prototyper         — 脚手架 + 最小实现
3. game-designer      — 评估原型
4. prototyper         — 编写发现报告
5. creative-director  — Go/No-Go 决策
6. producer           — 如通过则排期
```

## 委派权限矩阵

| 发起方 | 可委派给 |
|--------|---------|
| creative-director | game-designer, art-director, audio-director, narrative-director |
| technical-director | lead-programmer, devops-engineer, performance-analyst, technical-artist |
| producer | 任何 Agent（仅限其领域内的任务） |
| game-designer | systems-designer, level-designer, economy-designer |
| lead-programmer | gameplay-programmer, engine-programmer, ai-programmer, network-programmer, tools-programmer, ui-programmer |
| art-director | technical-artist, ux-designer, art-producer |
| audio-director | sound-designer |
| narrative-director | writer, world-builder |
| qa-lead | qa-tester |
| release-manager | devops-engineer, qa-lead |
| [引擎]-specialist | 对应引擎子专家 |

## 反模式（避免）

1. **跳层委派**：专家不得直接做主管的决策
2. **跨域实现**：Agent 不得修改非自己领域的文件
3. **影子决策**：所有决策必须文档化
4. **巨型任务**：每个任务应在 1-3 天内可完成
5. **假设驱动**：规格模糊时必须询问，而非猜测

# 项目规范

## 编码标准

- 所有游戏代码的公共 API 必须包含文档注释
- 每个系统必须有对应的架构决策记录（`docs/architecture/`）
- 玩法数值必须数据驱动（外部配置），禁止硬编码
- 所有公共方法必须可单元测试（依赖注入优于单例）
- 提交必须引用相关设计文档或任务 ID
- **验证驱动开发**：添加玩法系统时先写测试。UI 变更用截图验证

### 设计文档标准

- 所有设计文档使用 Markdown
- 每个机制有独立文档（`design/gdd/`）
- 必须包含 8 个必需章节：概述、玩家幻想、详细规则、公式、边界情况、依赖、调优旋钮、验收标准
- 平衡数值必须链接到源公式或理由

## 测试标准

### 按 Story 类型的测试证据要求

| Story 类型 | 必需证据 | 位置 | 门禁级别 |
|-----------|---------|------|---------|
| **逻辑**（公式、AI、状态机） | 自动化单元测试 — 必须通过 | `tests/unit/[系统]/` | 阻塞 |
| **集成**（多系统） | 集成测试或有文档的 Playtest | `tests/integration/[系统]/` | 阻塞 |
| **视觉/手感**（动画、VFX、手感） | 截图 + Lead 签核 | `production/qa/evidence/` | 建议 |
| **UI**（菜单、HUD、界面） | 手动走查文档或交互测试 | `production/qa/evidence/` | 建议 |
| **配置/数据**（平衡调优） | Smoke check 通过 | `production/qa/smoke-[日期].md` | 建议 |

### 自动化测试规则

- **命名**：文件 `[系统]_[功能]_test.[扩展名]`；函数 `test_[场景]_[预期]`
- **确定性**：每次运行产生相同结果 — 不用随机种子、不依赖时间
- **隔离**：每个测试自建自拆状态，不依赖执行顺序
- **不硬编码数据**：测试夹具使用常量文件或工厂函数（边界值测试除外）
- **独立性**：单元测试不调用外部 API、数据库或文件 I/O

### 不自动化的内容

- 视觉保真度（着色器输出、VFX 外观、动画曲线）
- "手感"品质（输入响应、感知重量、时机）
- 平台特定渲染（在目标硬件上测试，而非无头模式）
- 完整游戏流程（由 Playtest 覆盖）

### CI/CD 规则

- 每次推送到 main 和每个 PR 运行自动化测试套件
- 测试失败禁止合并 — 测试是 CI 中的阻塞门禁
- 禁止禁用或跳过失败测试来使 CI 通过 — 修复根本问题

**引擎特定 CI 命令**：
- Godot: `godot --headless --script tests/gdunit4_runner.gd`
- Unity: `game-ci/unity-test-runner@v4`（GitHub Actions）

## 协作协议

**用户驱动的协作，不是自主执行。**

每个任务遵循：**提问 → 选项 → 决策 → 草案 → 审批**

- Agent 必须在 Write/Edit 前询问"我可以写入 [文件路径] 吗？"
- Agent 必须在请求审批前展示草案或摘要
- 多文件变更需要完整变更集的明确审批
- 没有用户指令不提交

### 设计变更通知

设计文档变更时，game-designer 必须通知：
- lead-programmer（实现影响）
- qa-lead（测试计划更新）
- producer（进度影响评估）
- 相关专家 Agent

### 架构变更通知

ADR 创建或修改时，technical-director 必须通知：
- lead-programmer（代码变更）
- 所有受影响的专家程序员
- qa-lead（测试策略可能变化）
- producer（进度影响）

# 质量门禁体系

CCGS 通过 Director 门禁系统在各关键节点强制质量审查。

---

## 审查模式

通过 `production/review-mode.txt` 设置全局模式，也可每次运行时通过 `--review` 参数覆盖。

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **full** | 所有门禁都激活 | 团队协作、学习用户、需要全面反馈 |
| **lean**（默认） | 仅阶段门禁（`/gate-check`）激活 | 独立开发者、小团队 |
| **solo** | 所有门禁跳过 | Game Jam、原型、极速开发 |

```
# 检查逻辑（每个门禁执行前）：
1. 如果 Skill 带了 --review [mode] 参数 → 用该参数
2. 否则读取 production/review-mode.txt
3. 否则默认 full

solo → 跳过所有门禁
lean → 跳过非 PHASE-GATE 门禁
full → 正常执行所有门禁
```

---

## 判定结果

所有门禁返回三种结果之一：

| 结果 | 含义 | 默认处理 |
|------|------|---------|
| **APPROVE / READY** | 无问题，继续 | 推进工作流 |
| **CONCERNS [清单]** | 有问题但不阻塞 | 向用户展示，可选修订/接受/讨论 |
| **REJECT / NOT READY [阻塞项]** | 有阻塞问题 | 展示阻塞项，不得推进直到解决 |

**升级规则**：多个 Director 并行审查时，取最严格的结果（一个 NOT READY 覆盖所有 READY）。

## Tier 1 Director 门禁

### Creative Director（创意总监）— `creative-director`

| 门禁 ID | 名称 | 触发时机 |
|---------|------|---------|
| CD-PILLARS | 支柱压力测试 | 游戏支柱和反支柱定义后 |
| CD-GDD-ALIGN | GDD 支柱对齐检查 | 每个系统 GDD 编写后 |
| CD-SYSTEMS | 系统拆解愿景检查 | 系统索引完成后 |
| CD-NARRATIVE | 叙事一致性检查 | 叙事/世界观文档编写后 |
| CD-PLAYTEST | 玩家体验验证 | 测试报告生成后 |
| CD-PHASE-GATE | 阶段过渡创意就绪 | `/gate-check` 时 |

### Technical Director（技术总监）— `technical-director`

| 门禁 ID | 名称 | 触发时机 |
|---------|------|---------|
| TD-SYSTEM-BOUNDARY | 系统边界架构审查 | 系统依赖映射完成后 |
| TD-FEASIBILITY | 技术可行性评估 | 识别重大技术风险时 |
| TD-ARCHITECTURE | 架构签核 | 主架构文档完成后 |
| TD-ADR | 架构决策审查 | 单个 ADR 编写后 |
| TD-ENGINE-RISK | 引擎版本风险审查 | 使用截止日期后的引擎 API 时 |
| TD-PHASE-GATE | 阶段过渡技术就绪 | `/gate-check` 时 |

### Producer（制作人）— `producer`

| 门禁 ID | 名称 | 触发时机 |
|---------|------|---------|
| PR-SCOPE | 范围和时间线验证 | MVP 和时间线定义后 |
| PR-SPRINT | Sprint 可行性审查 | Sprint 计划确定前 |
| PR-MILESTONE | 里程碑风险评估 | 里程碑审查时 |
| PR-EPIC | Epic 结构可行性审查 | Epic 定义后 |
| PR-PHASE-GATE | 阶段过渡生产就绪 | `/gate-check` 时 |

### Art Director（美术总监）— `art-director`

| 门禁 ID | 名称 | 触发时机 |
|---------|------|---------|
| AD-CONCEPT-VISUAL | 视觉方向锚定 | 支柱锁定后 |
| AD-ART-BIBLE | 美术圣经签核 | 美术圣经完成后 |
| AD-PHASE-GATE | 阶段过渡视觉就绪 | `/gate-check` 时 |

## 阶段门禁并行协议

在 `/gate-check` 时，4 个 Director 并行审查：

```
同时启动（不等结果就发出所有调用）：
1. creative-director  → CD-PHASE-GATE
2. technical-director → TD-PHASE-GATE
3. producer           → PR-PHASE-GATE
4. art-director       → AD-PHASE-GATE

收集所有判定，应用升级规则：
- 任何 NOT READY/REJECT → 总判定最低 FAIL
- 任何 CONCERNS → 总判定最低 CONCERNS
- 全部 READY/APPROVE → 可判 PASS（仍需检查产物）
```

## 各阶段必选门禁

| 阶段 | 必选门禁 | 可选门禁 |
|------|---------|---------|
| Concept | CD-PILLARS, AD-CONCEPT-VISUAL | TD-FEASIBILITY, PR-SCOPE |
| Systems Design | TD-SYSTEM-BOUNDARY, CD-SYSTEMS, PR-SCOPE, CD-GDD-ALIGN（每个 GDD） | ND-CONSISTENCY, AD-VISUAL |
| Technical Setup | TD-ARCHITECTURE, TD-ADR（每个 ADR）, LP-FEASIBILITY, AD-ART-BIBLE | TD-ENGINE-RISK |
| Pre-Production | PR-EPIC, QL-STORY-READY（每个 Story）, PR-SPRINT, 四方阶段门禁 | CD-PLAYTEST |
| Production | LP-CODE-REVIEW（每个 Story）, QL-STORY-READY, PR-SPRINT（每个 Sprint） | PR-MILESTONE, QL-TEST-COVERAGE, AD-VISUAL |
| Polish | QL-TEST-COVERAGE, CD-PLAYTEST, PR-MILESTONE | AD-VISUAL |
| Release | 四方阶段门禁 | QL-TEST-COVERAGE |

## 添加新门禁

需要新门禁时：

1. 分配 ID：`[DIRECTOR-PREFIX]-[描述-SLUG]`（前缀：CD-、TD-、PR-、LP-、QL-、ND-、AD-）
2. 在 `.claude/docs/director-gates.md` 中添加完整定义
3. Skill 中通过 ID 引用，不内嵌完整提示词
